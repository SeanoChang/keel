# Keel v0.1 — Discord Channel Proxy

## Overview

Keel v0.1 is a thin Discord bot that proxies between a human and a running cubit instance. It has two jobs:

1. **Relay** — receive Discord messages, hand them to cubit as goals or quick commands
2. **Report** — poll cubit for state changes, post updates to Discord

Keel never touches Claude directly. Cubit owns the full agent lifecycle — identity (FLUCTLIGHT), memory (brief.md), planning, execution, workers, consolidation. Keel is purely the Discord skin.

## Architecture

```
You (Discord)
  → Keel (Go binary, long-running)
       ├── Discord listener (discordgo)
       ├── cubit mcp subprocess (persistent, JSON-RPC over stdio)
       └── Status poller goroutine (10s interval)
```

### Two Command Types

**Quick commands** (prefix: `!`) — Keel calls cubit MCP tools directly, posts result to Discord:

| Command | MCP Tool Call | Description |
|---------|--------------|-------------|
| `!status` | `cubit_status` | Queue health + brief token count |
| `!queue` | `cubit_queue {filter: "all"}` | List all tasks |
| `!queue pending` | `cubit_queue {filter: "pending"}` | List pending tasks |
| `!add <desc>` | `cubit_todo {description: "..."}` | Create a task |
| `!drain` | `cubit_drain` | Validate DAG, spawn executor |
| `!graph` | `cubit_graph {ascii: true}` | Visualize DAG |

**Full tasks** (no prefix, freeform message) — Keel submits a goal to cubit via MCP:

```
Discord: "research funding rate arb on BTC perpetuals"
  → Keel calls cubit_goal via MCP
  → Cubit loads identity + memory, spawns Claude session with MCP tools
  → Agent plans DAG (cubit_todo × N) → cubit_drain
  → Cubit runs workers, consolidation, lifecycle
  → Keel polls cubit_queue, reports state transitions to Discord
```

## Discord Visibility

All updates for one goal live in a single Discord thread to keep the channel clean. Keel posts the session message to the channel, then creates a thread on that message via `StartThreadWithMessage`. All subsequent updates go into the thread.

### Thread lifecycle

```
Channel:
  "⏳ Session: research funding rate arb on BTC perpetuals"
    └── Thread:
         "▶ Task 001 started: Research BTC perpetual funding rate mechanics"
         "▶ Task 002 started: Analyze historical funding rate data"
         "✓ Task 001 done: Identified 3 key funding rate patterns"
         "✓ Task 002 done: Collected 12 months of hourly data"
         "▶ Task 003 started: Backtest mean-reversion on funding spikes"
         "✓ Task 003 done: Sharpe 1.4, MaxDD -6%"
         "▶ Task 004 started: Summary — consolidate findings"
         "✓ Task 004 done: Final report written"
         "🎯 Drain complete. All tasks finished."
```

### Message formats

| Event | Format |
|-------|--------|
| Goal submitted | `⏳ Session: <first 80 chars of prompt>` |
| Task started | `▶ Task <id> started: <title>` |
| Task completed | `✓ Task <id> done: <summary>` |
| Task failed | `❌ Task <id> failed: <error>` |
| Drain complete | `🎯 Drain complete. All tasks finished.` |
| Quick command result | Formatted tool output |
| Error | `❌ <error message>` |

## Process Management

```
Keel (main process, long-running)
  │
  ├── Discord bot goroutine (discordgo)
  │
  ├── cubit mcp subprocess (persistent, JSON-RPC over stdio)
  │   └── spawned on startup, respawn if dies
  │   └── used for: quick commands + goal submission + status polling
  │
  └── Status poller goroutine
      └── activated after goal is submitted
      └── polls cubit_queue every 10s
      └── diffs task states, posts transitions to Discord thread
      └── stops when drain completes
```

### Lifecycle

1. Discord message arrives
2. If quick command (`!` prefix) → call cubit MCP tool → post result → done
3. If full task → call `cubit_goal` via MCP → post session started message + create thread
4. Start status poller → poll `cubit_queue` every 10s
5. Diff task states against last known → post transitions to thread
6. Drain complete = all tasks in "done" state and none "pending" or "active" (detected via `cubit_queue {filter: "all"}`) → post final message → stop poller

### Constraints

- **One active goal at a time.** If a freeform message arrives while drain is active, Keel replies "drain in progress, use !status to check." Quick commands (`!status`, `!queue`, `!graph`) still work during an active session. `!add` also works — it creates tasks in the existing queue.
- **Single channel.** Keel only listens and posts in the configured `channel_id`. Messages from other channels are silently ignored.
- **Allowed users only.** Messages from non-allowed users are silently ignored.
- **Keel never spawns Claude.** Cubit owns the full agent lifecycle.
- **Stateless.** Cubit owns all state. Keel tracks in-memory task states only for diffing during a session. If Keel restarts mid-drain, the thread is orphaned — cubit keeps running, but Keel cannot resume posting to the old thread. User can `!status` to check progress after restart.

## Configuration

```yaml
# ~/.config/keel/config.yaml
discord:
  token: "${DISCORD_BOT_TOKEN}"      # env var reference
  channel_id: "123456789"            # single channel Keel operates in
  allowed_users: ["987654321"]       # Discord user IDs that can send commands

cubit:
  binary: "cubit"                    # path to cubit binary
  project_dir: "/path/to/agent"     # cubit project root (where GOALS.md lives)

polling:
  interval: 10s                      # status poll frequency during drain
```

## Project Structure

```
keel/
├── main.go
├── go.mod
├── internal/
│   ├── discord/       # Discord bot, message handler, thread manager
│   ├── cubit/         # MCP client, JSON-RPC over stdio
│   └── config/        # Config loading
└── cmd/
    └── root.go        # keel serve (only command)
```

One command: `keel serve`. Starts the Discord bot, spawns cubit mcp, listens forever.

## Error Handling

| Scenario | Behavior |
|----------|----------|
| cubit mcp dies | Respawn, re-sync task snapshot from fresh `cubit_queue {filter: "all"}` call, log warning to Discord thread if mid-session. Quick commands retry once after respawn. |
| Discord disconnects | discordgo handles reconnect. Cubit keeps running — no work lost. |
| Goal submitted while drain active | Reply "drain in progress, use !status to check" |
| cubit_goal returns error | Post error to Discord, no retry |
| Drain never finishes | No timeout in v0.1 — user checks !status manually |
| Discord message >2000 chars | Truncate with "... (truncated)" suffix |
| Poll finds no state change | Skip posting, wait for next interval |

SIGINT/SIGTERM stops Keel. Cubit's drain workers are detached and survive Keel restarting.

## Dependencies

- `github.com/bwmarrin/discordgo` — Discord bot
- `gopkg.in/yaml.v3` — config parsing
- Standard library for everything else (exec, io, encoding/json, os/signal)

## Cubit Prerequisite

Keel v0.1 requires a `cubit_goal` MCP tool on the cubit side — an entry point that accepts a goal string, writes it to GOALS.md, and kicks off the planning + execution cycle. This is a cubit change, not a Keel concern.
